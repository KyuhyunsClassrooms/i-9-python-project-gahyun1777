# 최종 보고서

## 1. 프로그램 주제와 목적

내 프로그램의 주제: 반감기에 따른 체내 잔류 농도와 용량 계산기

이 프로그램을 만든 이유: 대수 시간에 심화탐구로 약물의 체내 잔류 용량을 계산하는 식을 조사하고 약물의 고유한 특성인 반감기를 잘알고 약물을 취급하는 것이 중요하다는 사실을 깨달았다. 또한 약물의 오남용이 초래하는 내성균의 문제가 심각하다는 사실을 알고 약물의 적절한 복용 주기 또한 중요하다는 것을 진로 시간의 조사를 통해 알았다. 따하서 반감기를 이용하여 체내 잔류 농도를 계산하고 적절한 복용 주기 추천을 해주는 프로그램을 만들었다.


---

## 2. 문제 분해 및 모델링

### 입력
약물 이름 (문자열): 데이터베이스에서 검색할 대상 (예: 타이레놀, 부루펜 등)

초기 복용량 (실수형): 사용자가 복용한 약물의 양 (mg 단위)

계산 방식 (문자열): 분석 기준선 모델링을 위한 '농도' 또는 '용량' 선택


### 처리
약물 데이터 검색: 입력된 이름을 바탕으로 2차원 리스트 drug_database를 순회하며 반감기와 최소 유효 농도(MEC) 데이터를 매칭합니다.

단위 통일 및 모델링: 선택한 계산 방식에 따라 상용 분포용적 상수를 활용해 연산 기준값을 변환합니다.
농도 방식: 초기 농도 = 복용량 / 분포용적, 기준치 = MEC
용량 방식: 초기 용량 = 복용량, 기준치 = MEC X 분포용적

시간 경과 시뮬레이션: while 반복문 속에서 1시간 단위로 시간이 흐를 때마다 수학적 반감기 공식을 적용해 체내 잔류량을 점진적으로 감소시킵니다.

약효 상태 판정 및 로그 누적: 매 시간 감소된 잔류량과 기준치를 비교하여 '🟢 효과 유지', '🔴 효과 소멸', '❌ 효과 없음' 등의 상태 조건문 판정을 내린 후 결과 2차원 리스트(hourly_log)에 시간별로 적립합니다.

### 출력
시간별 시뮬레이션 표: 0시부터 약효 소멸 시점까지(가독성을 위해 최대 24시간 제한)의 [시간, 체내 수치, 약효 상태] 구조를 깔끔하게 정렬된 표 형태로 출력합니다.

분석 리포트: 복용량이 부족하여 처음부터 약효가 없는지, 혹은 정상 작동 후 몇 시간 뒤에 효과가 소멸하는지 최종 결론 및 권고 메시지를 명확하게 보여줍니다.

## 3. 사용한 2차원 리스트 설명

2차원 리스트 이름: drug_database

한 행의 의미: 데이터베이스에 사전에 등록된 '약물 1종류에 대한 고유한 의약학적 특성 프로필'을 의미

각 열의 의미:

| 열 번호 | 의미 | 실제 예시 |
|---:|---|---|
| 0번 열 | 약물의 고유 명칭 (문자열) | "타이레놀" |
| 1번 열 | 약물의 생물학적 반감기 (정수형, 시간 단위) | 3 |
| 2번 열 | 약물의 최소 유효 농도 (MEC, 실수형) | 3.0 |
| 3번 열 |  |  |

이 2차원 리스트를 사용한 이유: 약물마다 가지고 있는 고유 정보(이름, 반감기, 최소 유효 농도)는 문자열, 정수, 실수로 서로 자료형이 다릅니다. 이 연관된 데이터를 하나의 묶음으로 구조화하여 관리하고, 반복문을 통해 사용자가 입력한 약물 이름을 탐색하며 필요한 의약학적 상수를 인덱스로 한 번에 꺼내오기에 가장 적합한 구조이기 때문입니다.

2차원 리스트 이름:hourly_log

한 행의 의미: 시뮬레이션이 진행되는 동안 매 1시간마다 계산되어 기록되는 '특정 시간대의 체내 약물 잔류 상태 및 판정 결과 데이터'를 의미

각 열의 의미:

| 열 번호 | 의미 | 실제 예시 |
|---:|---|---|
| 0번 열 | 복용 후 경과 시간 (정수형, 시간 단위) | 0 |
| 1번 열 | 해당 시간의 체내 잔류 수치 (실수형, mg 또는 mg/L) | 10.0 |
| 2번 열 | 해당 시간의 약효 상태 판정 결과 (문자열) | "🟢 효과 유지" (또는 "🔴 효과 소멸") |
| 3번 열 |  |  |

이 2차원 리스트를 사용한 이유: 시간 경과에 따라 실시간으로 변화하는 잔류량과 그에 따른 약효 상태 판정은 서로 긴밀하게 연결되어 저장되어야 합니다. while 반복문 안에서 매 시간 계산되는 3가지 데이터[시간, 수치, 상태]를 하나의 행으로 묶어 실시간으로 append하여 누적하고, 최종 출력을 담당하는 함수에 통째로 전달하여 깔끔한 표(Table) 형태로 정렬하여 화면에 시각화하기 위해 사용했습니다.

## 4. 함수 설명

| 함수 이름 | 역할 | 입력값 | 반환값 또는 출력 |
|---|---|---|---|
| find_drug(name) | 데이터베이스를 순회하며 일치하는 약물의 반감기와 MEC 정보를 탐색 | name (약물 이름) | (half_life, mec) 또는 None |
| simulate_mec(...) | 초기값에서 시작해 매 시간 반감기 감소 공식을 계산하여 기록 | half_life, mec, initial_val | hourly_log (2차원 리스트), wear_off_hour (정수) |
| display_result(...) | 생성된 2차원 로그 데이터를 가독성 좋은 표와 리포트로 구성하여 화면에 표시 | log_data, wear_off_hour, calc_type, drug_name | 정렬된 콘솔 텍스트 화면 출력 |

가장 중요한 함수 하나를 골라 자세히 설명하세요.

함수 이름: simulate_mec(half_life, mec, initial_val)

설명: 이 프로그램의 두뇌에 해당하는 핵심 시뮬레이션 엔진입니다. 처음에 들어온 초기 수치가 유효 기준치(mec)보다 낮으면 약효가 아예 없다고 판단하여 예외 로그와 함께 -1을 반환합니다. 기준치 이상일 때는 while 반복문이 실행되는데, 매 시간마다 체내 잔류 수치에 연속적인 감소율을 반영하기 위해 current_val * (0.5  (1 / half_life)) 연산을 수행합니다. 수치가 기준치 미만으로 떨어지는 순간 루프를 탈출하며 최종 '🔴 효과 소멸' 로그를 남기고, 총 유지 시간과 전체 2차원 리스트 데이터를 안전하게 반환합니다.

## 5. 조건문과 반복문 활용

### 조건문 사용 부분

어떤 조건을 검사했는가?
약물 등록 여부 판정: 사용자가 입력한 약물이 데이터베이스에 존재하는지 확인하기 위해 if drug_info is not None: 조건문을 사용했습니다.

초기 복용량 미달 검사: 약을 먹었으나 기준치보다 너무 낮아 아예 효과가 나타나지 않는 상태를 가려내기 위해 if current_val < mec: 검사를 수행했습니다.

24시간 출력 제한 분기: 시뮬레이션 결과가 너무 길어질 때 표의 가독성을 유지하기 위해 출력 루프 내에서 if row[0] > 24: 조건을 검사했습니다.


### 반복문 사용 부분

무엇을 반복 처리했는가?

선형 탐색 루프: find_drug 함수 내에서 고정 데이터베이스를 처음부터 끝까지 돌며 이름 매칭을 시도하는 for drug in drug_database: 구조를 활용했습니다.

시뮬레이션 타임 루프: 시간이 1시간씩 흐르는 연속적인 환경을 모완하기 위해 조건이 만족하는 동안 계속 도는 while current_val >= mec: 구조를 사용했습니다.

표 형식 순회 출력: 누적 완성된 2차원 결과 로그에서 한 줄씩 행을 꺼내 포맷팅에 맞춰 정렬 출력하는 for row in log_data: 문을 사용했습니다.


### 조건문과 반복문이 함께 작동한 부분
display_result 함수 내부에서 시뮬레이션 테이블을 그리는 영역입니다. for row in log_data: 반복문이 실행되면서 2차원 리스트의 요소를 하나씩 꺼내 화면에 출력하다가, 내부의 복합 조건문인 if row[0] > 24:를 만나는 순간 "24시간 이후 계산 생략"이라는 안내 메시지를 띄우고 break 명령어를 통해 반복문을 강제로 탈출합니다. 즉, 대량의 시뮬레이션 데이터 속에서 특정 조건이 만족할 때 반복 동작을 제어하도록 유기적으로 결합되어 작동합니다.

## 7. 테스트와 오류 수정

### 테스트 1: 정상 사례

입력: 부루펜 400 (계산 방식: 용량)

예상 결과: 부루펜의 반감기는 2시간, MEC는 2.0입니다. '용량' 방식이므로 기준 타깃 수치는 $2.0 \times 50.0 = 100\text{ mg}$이 됩니다. 초기 복용량 400mg은 2시간 뒤(2시) 200mg, 4시간 뒤(4시) 100mg이 되므로 정확히 4시까지 유지되다가 5시에 소멸해야 합니다.
 
실제 결과: 0시부터 4시까지 정상적으로 '🟢 효과 유지'가 기록되었고, 5시에 수치가 84.09mg으로 감소하며 '🔴 효과 소멸' 마크와 함께 '5시간 후 효과 소멸' 분석 문구가 정상 출력되었습니다.


### 테스트 2: 예외 또는 경계 사례

입력: 
약물 이름: 탁센

초기 복용량: 500 mg

계산 방식 선택: 농도

(당시 DB 데이터: 용량 MEC = 150mg, 농도 MEC = 15.0mg/L)

예상 결과: 똑같은 약물(탁센)을 똑같이 500mg 복용했으므로, 계산 방식이 '용량'이든 '농도'든 수학적으로 일치하는 결과(25시간 후 소멸)가 나와야 함.즉, 500mg을 먹었을 때 초기 농도 10.0가 유효 기준치를 넘어서 24시간 이상 약효가 유지되는 표가 출력되어야 정상임.

실제 결과: 약을 먹은 직후인 0시부터 '🔴 효과 소멸' 상태가 출력됨.

최종 분석 문구로 "탁센이(가) 처음 효과가 소멸되는 시기는 0시간 후입니다."라는 모순된 결과가 출력됨. (동일한 조건에서 '용량'을 선택했을 때는 25시간 동안 효과가 유지된다고 나옴)


### 발견한 오류와 수정한 방법

1. 발견한 오류 (Bug Analysis

오류 현상: 동일한 약물(탁센)과 동일한 초기 복용량(500mg)을 입력했음에도 불구하고, 계산 방식을 '용량'으로 선택했을 때는 25시간 동안 효과가 유지되는 반면, '농도'를 선택했을 때는 0시부터 즉시 '🔴 효과 소멸'이 뜨는 모순이 발생했습니다.

원인 분석:
1. DB 기준치의 불일치: 데이터베이스(drug_database)의 2번 열에 저장된 탁센의 기본 수치는 3.0이었습니다. 코드의 주석과 흐름상 이는 농도 기준(3.0 mg/L)을 의미합니다.

2. 잘못된 '용량' 변환 공식: 기존 코드에서는 사용자가 '용량'을 선택했을 때 target_mec = mec_concentration * VOLUME_OF_DISTRIBUTION 구조를 사용했습니다. 탁센의 경우 $3.0 \times 50.0 = 150.0\text{ mg}$가 되어 정상적인 용량 MEC 기준치로 작동했습니다.

3. '농도' 선택 시 초기값 계산 오류: 사용자가 '농도'를 선택했을 때 초기 투여 농도는 $\text{복용량}(500\text{ mg}) / \text{분포용적}(50.0\text{ L}) = 10.0\text{ mg/L}$가 됩니다. 이 값($10.0$)은 탁센의 실제 농도 MEC인 3.0보다 크기 때문에 정상적으로 효과가 유지되어야 합니다. 그러나 코드 내부 유효성 검증이나 변수 매칭 과정에서 초기 투여 농도($10.0$)가 엉뚱하게 변환된 기준치나 매우 높은 임의의 수치와 비교되면서, 시작하자마자 MEC 미달로 판정되어 0시부터 효과 소멸 결과가 출력되는 농도/용량 간의 논리적 불일치 오류가 발생했습니다.

2. 수정한 방법 (Resolution)

수정 알고리즘 설계: 계산 방식('농도' 또는 '용량')에 상관없이, 비교의 기준이 되는 체내 현재 수치(current_val)와 최소 유효 농도 수치(target_mec)의 단위를 완벽히 일치시키도록 조건문 연산 구조를 명확하게 재정비했습니다.

코드 수정 내용:

[농도 선택 시]: 초기 복용량(mg)을 분포 용적(L)으로 나눈 값(initial_dose / VOLUME_OF_DISTRIBUTION)을 초기값으로 설정하고, 목표치(target_mec)는 DB에 저장된 원래의 농도 MEC 값(mec_concentration)을 그대로 사용하도록 고정했습니다.

[용량 선택 시]: 초기 복용량(mg)을 그대로 초기값으로 사용하고, 목표치(target_mec)는 DB의 농도 MEC에 분포 용적을 곱한 값(mec_concentration * VOLUME_OF_DISTRIBUTION)으로 일관성 있게 변환되도록 메인 흐름(__main__)의 if-else 제어구조와 변수 대입 논리를 전면 수정했습니다.

수정 후 결과: 탁센 500mg 복용 시 '농도'를 선택하더라도 초기 농도 $10.0\text{ mg/L}$가 기준치인 $3.0\text{ mg/L}$ 이상임을 올바르게 인식하게 되었으며, '용량' 선택 시와 수학적으로 완전히 일치하는 '25시간 후 효과 소멸'이라는 정상적인 시뮬레이션 표와 분석 리포트가 출력됨을 확인했습니다.

## 8. Gemini 활용 내용 요약

Gemini에게 도움받은 부분:
단순히 반감기 횟수대로 뚝뚝 떨어지는 거친 연산 대신, 1시간 단위로 체내 잔류량이 부드럽고 매끄럽게 곡선을 그리며 감소하도록 만들어 주는 지수 법칙 기반의 시간당 감소율 공식($현재값 \times 0.5^{\frac{1}{반감기}}$)을 정립하는 데 큰 가이드를 받았습니다.

Gemini가 제안했지만 내가 수정하거나 사용하지 않은 부분:
Gemini는 최초 설계 상담 시 데이터베이스에 약물이 없을 경우 사용자에게 콘솔로 즉석에서 반감기와 MEC를 새로 입력받아 drug_database.append() 방식으로 실시간 추가하는 대화형 알고리즘을 추천해 주었습니다. 하지만 터미널 창에서 매번 수동으로 추가하는 것은 프로그램 종료 시 데이터가 소멸되어 실효성이 낮고 예외 입력 처리가 지나치게 복잡해진다고 판단하여, 과감히 깔끔한 예외 메시지를 띄우고 시스템을 세련되게 차단하는 구조로 변경했습니다.

최종 코드에서 내가 직접 판단하거나 수정한 부분:
탁센(반감기 14시간)처럼 대사 속도가 매우 느린 약물을 시뮬레이션하면 콘솔창에 수십 줄의 텍스트가 도배되어 가독성이 떨어지는 문제를 직접 발견했습니다. 이를 막기 위해 출력 제어 함수(display_result) 내부에 24시간 제한용 조건 탈출문(if row[0] > 24: break)을 독자적으로 빌드하여 결과 창의 UI 정렬 상태를 직관적이고 콤팩트하게 다듬었습니다.

## 9. 한계와 개선 방안

현재 프로그램의 한계:
본 프로그램은 한 번 약을 복용하고 끝나는 '단회 복용' 모델만 지원합니다. 그러나 실제 환자들은 약효가 소멸하기 전 일정 시간(예: 4~6시간)마다 약을 연속적으로 2차, 3차 추가 복용하는 경우가 많습니다. 현재의 단순 while문 구조로는 체내에 기존 잔류량과 새 복용량이 누적되어 융합되는 복합적인 정상상태(Steady-state) 중첩 농도를 계산해 내지 못합니다.

다시 개선한다면 추가하고 싶은 기능:
연속 중복 복용 스케줄러: 사용자가 "6시간 간격으로 300mg씩 총 3회 복용"과 같이 복약 패턴을 입력하면, 시간이 흐름에 따라 톱니바퀴 모양으로 체내 농도가 축적 및 대사되는 누적 농도 그래프 시뮬레이션 기능을 추가하고 싶습니다.

외부 데이터 파일 입출력(I/O): 프로그램 내부 하드코딩 리스트 구조를 탈피하여 외부 CSV나 JSON 파일과 실시간 연동해 사용자가 직접 새로운 약물 데이터를 안전하게 추가/삭제하고 영구 저장할 수 있는 영속성 데이터 관리 기능을 도입하고 싶습니다.
